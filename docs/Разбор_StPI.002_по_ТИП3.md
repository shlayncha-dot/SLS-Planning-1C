# Разбор детали `StPI.002` по алгоритму верификации ТИП 3

## 1) Выбор типа алгоритма
1. Для детали берётся префикс до первой точки: `StPI`.
2. В настройках верификации `StPI` входит в условие для **ТИП 3** (`StFO, StSwO, StS, StP, StPI, StBM, StBO`).
3. Значит для детали применяется `ApplyType3Algorithm(...)`.

## 2) Что именно делает ТИП 3
Алгоритм ТИП 3 пытается извлечь ключ поиска через регулярное выражение формата:

- `prefix.rest.suffix` + опционально ревизия (`.R...`) + опциональный хвост (`-...`).

Ключевые части:
- `prefix` — до первой точки;
- `rest` — обязательная средняя часть между двумя точками;
- `suffix` — обязательный числовой хвост (`ddd.ddd`, `ddd.dd`, `dd.ddd`, `ddd`, `dd`);
- `revision` — опционально, в формате `.R...`.

## 3) Пошаговый разбор `StPI.002`
Строка: `StPI.002`

1. `prefix = StPI` — выделяется.
2. Далее алгоритм ожидает формат **`prefix.rest.suffix`** (то есть минимум две точки в полном обозначении).
3. В `StPI.002` есть только одна точка, поэтому обязательный блок `rest` отсутствует.
4. Из-за этого извлечение `Type3SearchKey` завершается с результатом `null`.
5. `ApplyType3Algorithm(...)` при `null` возвращает пустой список совпадений.

## 4) Результат верификации для этой строки
Для `StPI.002` по текущему ТИП 3:
- кандидаты PDF не находятся алгоритмом ТИП 3;
- итоговый статус проверки для строки будет как «не найдено» (`missing`), если строка участвует в PDF-верификации и не подпадает под правило `-WD`.

## 5) Почему это важно
Сейчас ТИП 3 в коде рассчитан на «трёхчастную» структуру (`prefix.rest.suffix`).
Короткое обозначение вида `StPI.002` в эту структуру не попадает, даже если префикс корректно относит деталь к ТИП 3.
